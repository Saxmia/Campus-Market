# 校园二手交易平台数据库系统

本项目是为数据库大作业设计的“校园二手交易平台”，基于 **Next.js 14** 和 **Vercel Postgres (PostgreSQL)** 构建，实现了完整的数据库设计、数据操作及 Web 交互。

## 2. 核心功能实现说明

### 数据库架构 (Database Schema)
- **User (用户表)**: 存储用户信息 (`user_id`, `user_name`, `phone`)。
- **Item (商品表)**: 存储二手商品 (`item_id`, `item_name`, `category`, `price`, `status`, `seller_id`)。
- **Orders (订单表)**: 存储交易记录 (`order_id`, `item_id`, `buyer_id`, `order_date`)。
- **视图 (Views)**:
    - `sold_items_view`: 展示已售商品名及买家 ID。
    - `unsold_items_view`: 展示所有未售出的商品。

### 实现的 SQL 操作
- **基本操作**: 插入新商品、修改价格、删除未售商品。
- **基本查询**: 未售商品查询、价格 > 30 查询、特定分类查询。
- **连接查询**: 查询已售商品及买家姓名、查询订单详情。
- **聚合查询**: 统计商品总数、分类统计、平均价格、最活跃卖家。
- **业务逻辑**: 购买操作使用 **SQL 事务**，确保新增订单与修改商品状态原子化完成。

## 4. 简答题回答

### (八) 安全性
1. **如何防止普通用户删除数据？**
   - **权限校验**: 在 API 层通过鉴权机制（如 JWT）识别用户身份，非管理员或非本人禁止调用 DELETE 接口。
   - **数据库角色**: 在数据库层创建不同权限的角色，Web 应用使用仅具有 `SELECT/INSERT/UPDATE` 权限的角色。
2. **如何限制用户只能查询数据？**
   - **接口限制**: 只对外暴露查询类的 GET 接口。
   - **只读用户**: 配置数据库只读账号供查询服务使用。

### (九) 并发与恢复
1. **两个用户同时购买同一商品会出现什么问题？**
   - **竞态条件 (Race Condition)**: 若不加控制，可能出现两个用户同时读取到商品为“未售出”状态，并各自生成订单，导致一物多卖。
2. **如何解决（例如加锁/事务）？**
   - **数据库事务**: 将“检查状态”、“插入订单”、“更新状态”包装在 `BEGIN...COMMIT` 事务中。
   - **行级锁**: 在查询商品状态时使用 `SELECT ... FOR UPDATE`，锁定该行直至事务结束，确保其他事务无法同时修改该商品。
3. **如果系统崩溃，如何恢复订单数据？**
   - **WAL 日志**: 利用 PostgreSQL 的预写式日志（Write-Ahead Logging），系统重启后可自动重做已提交的事务。
   - **备份恢复**: 定期进行全量备份（pg_dump）及增量日志备份，通过日志点恢复（PITR）将数据恢复到崩溃前的状态。
